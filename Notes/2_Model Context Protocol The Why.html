04 · What is Context?

Definition

Context = everything an AI can "see" when it generates a response.

More formally: the information (conversation history, external docs, etc.) that the LLM uses to generate a response.

Example: while chatting with ChatGPT, the past messages form the context.

Software Engineer Example — Context is Scattered

A dev's context lives across multiple tools simultaneously:

Jira (tickets) GitHub (code) Slack (discussions) Google Drive (docs) MySQL (data)

→ ChatGPT only sees what you paste. Context is scattered.

05 · The Copy-Paste Hell

Problems Caused by Scattered Context

  • Need to paste thousands of lines to ask one simple question
  • Developers became Human APIs — manually bridging tools and AI
  • Context Assembly Time > Development Time

  • Scaling problems — gets worse with larger projects

07 · Problem with Tools

The N × M Integration Problem

Every AI agent (Chatbot, Coding Agent, Analytics Agent) needs custom integration with every tool (Jira, Slack, GitHub, MySQL).

N agents × M tools = Development Nightmare

  • Different authentication methods per tool
  • Different data formats and API patterns
  • Different error handling approaches

Additional Problems

  • Maintenance problem — every API change breaks integrations
  • Security fragmentation — each integration handles auth differently
  • Cost and time wastage — rebuilding the same integrations repeatedly
08 · Enter MCP — The Solution

Before MCP

N agents × M tools = N×M custom integrations

Each agent writes its own GitHub connector, Jira connector, Slack connector…

After MCP

N clients + M servers = M+N integrations only

GitHub builds ONE MCP Server. Any AI tool just connects to it via standard protocol.

MCP Architecture

ChatGPT / Claude / Cursor→ MCP Client SDK→ MCP Server SDK (GitHub)
MCP Client SDK→ MCP Server SDK (Google Drive)
MCP Client SDK→ MCP Server SDK (Any Tool…)

The Client has to just connect to the server using the same language (MCP).

Server Does the Heavy Lifting

The MCP Server handles all complexity so the client doesn't have to:

  • Authentication with the external service (e.g. GitHub)
  • API rate limiting
  • Data format translation
  • Error handling
  • Service-specific business logic
09 · MCP vs Function Calling vs API

Function / Tool Calling (old way)

Developer writes their own function that calls an external API:

def get_weather(city: str): url = f"https://myweatherapi.com/weather?city={city}&key=API_KEY" response = http_get(url) data = json_parse(response) return f"{city}: {data['temp_c']}°C, {data['condition']}" # Server side (Flask route) @app.route("/weather") def weather_endpoint(): city = request.args.get("city") result = lookup_weather(city) return jsonify(result)

MCP Way

Client just calls tools via MCP protocol — no custom HTTP code needed:

# Client — simple result = call_tool("get_weather", {"city": "London"}) # Server — registers tools using MCP SDK from mcp.server import Server server = Server("WeatherServer") @server.tool("get_weather") def get_weather(city: str): data = lookup_weather(city) return {"temperature": data["temp_c"], "condition": data["condition"]} server.start()
10 · Benefits & MCP Ecosystem

Benefits of MCP

  • N clients + M servers = M + N integrations (not N×M)
  • No maintenance overhead
  • Reduced cost and time
  • Better security — standardised auth

MCP Ecosystem — Network Effect

  • More AI Chatbots support MCP → More valuable for services to build MCP servers
  • More MCP servers available → More valuable for AI tools to support MCP
  • More adoption → More standardization → More ecosystem value
  • Not supporting MCP = being cut off from a rapidly growing ecosystem
