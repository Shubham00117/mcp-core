What is MCP?

MCP (Model Context Protocol) is a standardized protocol that allows AI models (LLMs) to connect to external tools and data sources through a common interface. It decouples the AI host from the services it uses.

02

Architecture Components

🖥️ Host

The AI application (e.g., Claude Desktop, ChatGPT). Contains one or more Clients. Talks to the LLM and manages user interactions.

🔌 Client

Lives inside the Host. Each Client maintains a 1-to-1 connection with exactly one Server. Sends requests and receives responses.

⚙️ Server

An external program (GitHub, Slack, Filesystem, etc.) that exposes tools, resources, and prompts to the Host via JSON-RPC.

Key Rule

One Host → many Clients → many Servers. Each Client connects to exactly one Server. The Host manages all Clients internally.

03

Benefits of MCP

Decoupling

Safety — Servers are isolated; a faulty server can't crash the host.
Parallelism — Multiple clients can call multiple servers simultaneously.

Scalability

Add new servers without changing the Host.
Any LLM can connect to any MCP server using the same protocol.
04

MCP Primitives

Primitives are the things a Server can offer to the Host. Every MCP server exposes some combination of these three building blocks.

Tools

Actions the AI asks the server to perform. e.g. create issue, list PRs, send message.

Resources

Structured data sources the AI can read. e.g. file contents, repo data, open issues list.

Prompts

Predefined prompt templates the server offers to shape AI behavior. Ensures consistent, structured output.

Standard Operations per Primitive

Primitive Operations Meaning
Tools tools/list
tools/call		 "What tools do you have?" / "Run this tool with these args."
Resources resources/list
resources/read
resources/subscribe 		"What's available?" / "Give me this content." / Subscribe to updates.
Prompts prompts/list
prompts/get		 "What templates do you have?" / "Fetch this template."

Prompts Primitive — Why It Matters

Without a prompt template, the AI writes vague issues:

// ❌ Too vague
Title: Login bug
Body: The login button doesn't work.

With an issue_report_prompt template from the server:

// ✅ Structured & useful
Title: Bug — Login Button Not Working

**Steps to Reproduce:**
1. Open the login page
2. Enter valid credentials
3. Click the Login button

**Expected:** User logged in & redirected to dashboard
**Actual:** Nothing happens
**Environment:** Chrome 121, macOS 14.2
05

MCP Data Layer

What is the Data Layer?

The shared language and grammar that all MCP components agree to use when communicating. In MCP, JSON-RPC 2.0 is the foundation of the data layer.

06

JSON RPC 2.0

JSON-RPC combines Remote Procedure Calls (RPC) with the simplicity of JSON, allowing structured requests and responses in a standardized format.

Request Structure

{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": { ... },
  "id": 1
}

Success Response

{
  "jsonrpc": "2.0",
  "result": { ... },
  "id": 1
}

Error Response

{
  "jsonrpc": "2.0",
  "error": {
    "code": -32601,
    "message": "Method not found"
  },
  "id": 1
}

Why JSON-RPC for MCP?

Lightweight — minimal overhead, pure JSON
Bi-directional — client→server and server→client
Transport-agnostic — works over STDIO or HTTP/SSE
Supports Batching — send multiple requests in one call
Supports Notifications — fire-and-forget server→client messages (no id field)
07

JSON-RPC in Action — Examples

1. Discover Tools — tools/list

Request

{ "jsonrpc": "2.0",
  "id": 1, "method": "tools/list",
  "params": {} }

Response

{ "result": { "tools": [
  { "name": "github.listIssues" },
  { "name": "github.listPulls" }
] } }

2. Call a Tool — tools/call

Request

{ "method": "tools/call",
  "params": {
    "name": "github.listIssues",
    "arguments": {
      "owner": "ownerName",
      "repo": "repoName"
    }
  } }

Response

{ "result": {
  "content": [
    { "id": 271,
      "title": "Crash on startup",
      "state": "open" },
    { "id": 272,
      "title": "Null pointer bug" }
  ] } }

3. List & Read Resources — resources/list & resources/read

List → returns URIs

{ "resources": [
  { "uri": "github://repo/owner/repo",
    "name": "Main repo" },
  { "uri": "github://repo/.../issues",
    "name": "Open issues" }
] }

Read → fetches content by URI

// Request passes URI
{ "method": "resources/read",
  "params": { "uri": "github://.../issues" } }

// Response returns content
{ "contents": [
  { "uri": "...", "text": "[{...issues...}]" }
] }

4. Batching — many in one request

Send an array of requests; get an array of responses back.

// Request (array)
[ { "id": 5, "method": "tools/call",
    "params": { "name": "github.listIssues" } },
  { "id": 6, "method": "tools/call",
    "params": { "name": "github.listPulls" } } ]

// Response (array)
[ { "id": 5, "result": { ... } },
  { "id": 6, "result": { ... } } ]

5. Notification (fire-and-forget)

Server → Client push. No id field = no response expected.

// Server sends this; client just listens
{
  "jsonrpc": "2.0",
  "method": "files/updated",
  "params": {
    "fileId": "abc123",
    "name": "ProjectPlan.docx",
    "updatedBy": "alice@example.com"
  }
}

6. Error Example

Missing required field 'repo':

{ "error": { "code": -32602,
  "message": "Missing required field 'repo'" } }
08

MCP Transport Layer

The Transport Layer is the mechanism that physically moves JSON-RPC messages between Client and Server. The choice of transport depends on the type of server (local or remote).

Local Server STDIO

Program running on your own machine. Uses Standard I/O streams to communicate.

Remote Server HTTP + SSE

Program running on another computer over the network or internet.

09

Local Servers — STDIO Transport

What is STDIO?

STDIO refers to the built-in streams every program has:

stdin — the input the program reads from
stdout — the output the program writes to
In MCP, these streams are used as the transport layer between the Client and Server.

How STDIO Works

Host launches the server as a subprocess on the same machine.
Host (client) writes JSON-RPC messages into the server's STDIN.
Server reads, processes, and writes responses back to its STDOUT.

Benefits of STDIO

Fast — data passed directly between processes, no network hop.
Secure — no open network port; communication is fully local.
Simple — every language/runtime supports stdin/stdout. No extra libraries needed.
10

Remote Servers — HTTP + SSE Transport

HTTP

Allows the Host to reach servers running anywhere on the internet.
Host sends JSON-RPC requests using HTTP POST with a JSON payload.
Supports standard HTTP auth methods like API keys.

SSE — Server Sent Events

SSE is an extension of HTTP that enables server-to-client streaming.

Server sends multiple messages to the client over a single open connection.
Instead of one large JSON blob, the server streams chunks as they become ready.
Ideal for long-running tasks or incremental updates (e.g., AI generation in progress).
