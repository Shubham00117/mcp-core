MCP Lifecycle Overview

The MCP Lifecycle describes the complete sequence of steps that govern how a Host (client) and a Server establish, use, and end a connection during a session.

02

Initialization Phase

The initialization phase MUST be the first interaction between client and server. It establishes protocol version compatibility and exchanges/negotiates capabilities.

Step 1 — Client → Server

Client sends initialize request

{
  "jsonrpc": "2.0", "id": 1, "method": "initialize",
  "params": {
    "protocolVersion": "2025-03-26", // MCP protocol version
    "capabilities": {
      "roots": { "listChanged": true }, // client capabilities
      "sampling": {}
    },
    "clientInfo": { "name": "IDEPlugin", "version": "1.0.0" } // client info
  }
}
Step 2 — Server → Client

Server replies with its own capabilities

{
  "jsonrpc": "2.0", "id": 1, "result": {
    "protocolVersion": "2025-03-26", // negotiated version
    "capabilities": {
      "tools": { "listChanged": true },
      "resources": { "listChanged": true, "subscribe": true }
    },
    "serverInfo": { "name": "FileSystemServer", "version": "2.5.1" },
    "instructions": "Server is ready to accept commands"
  }
}
Step 3 — Client → Server

Client sends initialized notification

{ "jsonrpc": "2.0", "method": "notifications/initialized" }

✓ Client and Server are now connected — normal operations can begin.

Rule 1: The client SHOULD NOT send requests (other than pings) before the server has responded to the initialize request.
Rule 2: The server SHOULD NOT send requests (other than pings and logging) before receiving the initialized notification.
03

Version Negotiation

Client proposes its preferred protocol version. Server replies with the highest version it supports from the supported list.

// Client proposes
"protocolVersion": "2025-03-26"

// Server responds with what it supports
"protocolVersion": "2024-11-05"

// Supported versions on both sides
SUPPORTED_PROTOCOL_VERSIONS = ["2025-03-26", "2024-11-05"]
04

Capability Negotiation

Client and server capabilities establish which protocol features will be available during the session. Only negotiated capabilities may be used.

Side Capability Sub-capabilities
Client roots listChanged
Client sampling —
Client elicitation —
Server prompts listChanged
Server resources listChanged, subscribe
Server tools listChanged
Server logging —
05

Operation Phase

During the operation phase, client and server exchange messages according to the negotiated capabilities. Must respect the negotiated protocol version and only use successfully negotiated capabilities.

Capability Discovery (right after initialized)

Immediately after the initialized notification, the client sends discovery requests:

// Client discovers available tools
{ "id": 1, "method": "tools/list" }

// Client discovers available prompts
{ "id": 2, "method": "prompts/list" }

// Client discovers available resources
{ "id": 3, "method": "resources/list" }

// Note: prompts/list and resources/list may return -32601 if server doesn't support them
06

Example: Filesystem Server Tools

Tools discovered via tools/list on the secure-filesystem-server:

read_fileRead complete contents of a single file. Only works within allowed directories.
read_multiple_filesRead multiple files simultaneously. More efficient than reading one by one.
write_fileCreate a new file or overwrite an existing file. Requires: path + content.
edit_fileMake line-based edits to a text file. Returns a git-style diff. Uses oldText + newText.
create_directoryCreate a directory or ensure it exists. Can create nested directories in one operation.
list_directoryGet a detailed listing with [FILE] and [DIR] labels. Works within allowed directories.
directory_treeGet a recursive tree view of files/directories as JSON structure.
move_fileMove or rename files/directories. Requires: source + destination paths.
search_filesRecursively search for files and directories. Case-insensitive. Returns full paths.
list_allowed_directoriesReturns directories this server can access. Call this first before accessing files.

Tool Calling

Request

{
  "jsonrpc": "2.0", "id": 3,
  "method": "tools/call",
  "params": {
    "name": "getFile",
    "arguments": {
      "owner": "campusx-official",
      "repo": "mcp-examples",
      "path": "README.md",
      "ref": "main"
    }
  }
}

Response

{
  "jsonrpc": "2.0", "id": 3,
  "result": {
    "content": "# MCP Examples\nThis repo...",
    "encoding": "utf-8",
    "sha": "f3c0..."
  }
}

08

Shutdown Phase

One side (typically the client) initiates shutdown.
No special JSON-RPC shutdown message is defined — the transport layer handles termination.

🖥️ Shutdown in STDIO

Client-initiated (SHOULD):

Close input stream to the child process (server)
Wait for server to exit
Send SIGTERM if server does not exit in time
Send SIGKILL if still unresponsive

Server-side (SHOULD):

Close output stream to the client → Exit process

🌐 Shutdown in HTTP

Client-initiated (common):

The client (Host) closes the HTTP connection(s) it opened to the Server.

Server-initiated (possible):

Server may close the connection from its side.
Client must be prepared to detect a dropped connection and handle it (e.g., reconnect).
09

Pings

Ping is a lightweight request/response method defined in MCP. Its purpose is to check whether the other side (Host or Server) is still alive and the connection is responsive.

Ping Format

// Request (either side can ping)
{ "jsonrpc": "2.0", "id": 42, "method": "ping" }

// Response (empty result)
{ "jsonrpc": "2.0", "id": 42, "result": {} }

When is Ping used?

Allowed before full initialize — to check if the other side is up.
Sent periodically when there's no activity — prevents connection being silently dropped by OS, proxies, or firewalls.
10

Error Handling

Error handling in MCP is how the Host (client) and Server signal that something went wrong with a request. MCP inherits JSON-RPC's standard error object format.

Error Object Structure

{
  "jsonrpc": "2.0", "id": 5,
  "error": {
    "code": -32601, // integer
    "message": "Method not found", // human-readable
    "data": { "extra": "optional info" } // optional
  }
}

Causes of Error

Unsupported or mismatched protocol version
Calling a method for a capability that wasn't negotiated
Invalid arguments passed to a tool
Internal server failure while processing a request
Timeout exceeded → client cancels request
Malformed JSON-RPC messages

Common Error Codes

Code Name Meaning Example
-32601 Method not found Called a method that doesn't exist or wasn't advertised Host calls prompts/list but server never advertised prompts
-32602 Invalid params Request sent with wrong or missing parameters Tool expects "path" but client sends "file"
-32600 Invalid Request Malformed JSON-RPC request structure Missing required fields like jsonrpc or method
-32700 Parse error JSON could not be parsed Request body is not valid JSON
-32000+ Server-defined errors Custom errors defined by the server Auth failure, rate limit exceeded, quota errors
11

Timeout

Timeout ensures requests don't hang forever. It protects against unresponsive or overloaded servers and ensures resources (memory, CPU) aren't held indefinitely.

Purpose

Protects against unresponsive or overloaded servers
Ensures resources (memory, CPU) aren't held indefinitely
Gives user feedback instead of waiting forever

How It Works

SDKs let Client set a per-request timeout (e.g., 30s)
If deadline passes with no result → client triggers timeout
Client sends a cancellation notification to tell server to stop
Server must stop processing and must NOT return a result
12

Cancellation

When a timeout occurs, the client sends a notifications/cancelled message referencing the original request ID and the reason:

// Original request (includes progressToken for tracking)
{ "id": "7", "method": "tools/call",
  "params": { "name": "searchCode", "arguments": { "query": "MCP" },
              "_meta": { "progressToken": "tok-7" } } }

// Cancellation notification (fire-and-forget, no id)
{ "jsonrpc": "2.0", "method": "notifications/cancelled",
  "params": { "requestId": "7", "reason": "Timeout exceeded (30s)" } }
13

Progress Notification

Purpose: Let the client know a long-running request is still making progress. The client includes a progressToken in the request's _meta, and the server sends notifications/progress updates while working.

Request (with progressToken)

{ "id": 7, "method": "tools/call",
  "params": {
    "name": "searchCode",
    "arguments": { "query": "MCP lifecycle" },
    "_meta": { "progressToken": "tok-7" }
  } }

Server Progress Update

{ "method": "notifications/progress",
  "params": {
    "progressToken": "tok-7",
    "progress": 60,
    "total": 100,
    "message": "Searching 600 of 1000 files"
  } }

Full Session Flow at a Glance

CLIENT → SERVER  initialize (version + capabilities + clientInfo)
SERVER → CLIENT  Result (version + server capabilities + serverInfo)
CLIENT → SERVER  notifications/initialized ← connection established
CLIENT → SERVER  tools/list, prompts/list, resources/list
SERVER → CLIENT  Tool/prompt/resource schemas (or -32601 if unsupported)
— — — — — OPERATION PHASE — — — — —
CLIENT → SERVER  tools/call {name, arguments, _meta.progressToken}
SERVER → CLIENT  notifications/progress (while working)
SERVER → CLIENT  Result or Error response
— — — — — SHUTDOWN — — — — —
CLIENT  Closes transport (STDIO stream / HTTP connection)
