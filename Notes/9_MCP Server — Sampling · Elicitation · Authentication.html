1. Sampling

Servers asking the LLM to generate text — server-initiated AI calls

CONCEPT

What is Sampling?

💡

Definition: Sampling lets an MCP server request the client's LLM to generate text. Instead of the client always calling the server — the server calls back to the LLM for AI completions. This enables agentic loops where the server can "think" using the LLM.

Normal Flow vs Sampling Flow
NORMAL (client → server)
LLM / Client App
↓ tool call
MCP Server
↓ result
LLM / Client App
SAMPLING (server → LLM)
LLM / Client App
↓ tool call
MCP Server
↑ sampling request
LLM generates text
↓ completion back
Server uses result
Human in the loop — client can review/approve LLM requests before execution

Why use Sampling?

  • Server needs LLM to summarise, classify, or generate data mid-tool
  • Build agentic workflows inside the server
  • Multi-step reasoning — server can chain LLM calls
  • Server doesn't need its own LLM API key

Key Constraint

  • Only available in local (STDIO) servers that support it
  • Client must explicitly grant sampling permission
  • Human-in-the-loop — client reviews the request
  • Not all MCP clients support sampling
PARAMS

Sampling Request Parameters

messages

List of conversation messages (role + content) sent to the LLM for completion.

maxTokens

Maximum number of tokens the LLM should generate in the response.

systemPrompt

Optional system prompt to guide the LLM's behaviour for this specific request.

temperature

Controls randomness of the output. 0 = deterministic, 1 = creative.

stopSequences

Array of strings where generation should stop.

modelPreferences

Hints about which model to use (cost, speed, intelligence priorities).

CODE

Sampling in FastMCP

sampling_server.pyPython
from fastmcp import FastMCP
from fastmcp.sampling import SamplingMessage

mcp = FastMCP("SamplingDemo")

@mcp.tool()
async def summarise_text(text: str, ctx) -> str:
    """Use the LLM to summarise the provided text."""

    # Server requests the client's LLM to generate a summary
    result = await ctx.sample(
        messages=[
            SamplingMessage(
                role="user",
                content=f"Please summarise this in 2 sentences:\n\n{text}"
            )
        ],
        system_prompt="You are a concise summarisation assistant.",
        max_tokens=100
    )

    return result.content.text


@mcp.tool()
async def classify_expense(description: str, ctx) -> str:
    """Ask LLM to classify an expense into a category."""

    result = await ctx.sample(
        messages=[
            SamplingMessage(
                role="user",
                content=f"Classify this expense: '{description}'. "
                         "Reply with ONLY one word: Food/Travel/Shopping/Bills/Other"
            )
        ],
        max_tokens=10,
        temperature=0      # deterministic
    )

    return result.content.text.strip()


if __name__ == "__main__":
    mcp.run()   # STDIO transport (required for sampling)
⚠️

ctx parameter: The tool function must accept ctx as a parameter. This is the MCP context object that provides access to ctx.sample(), ctx.log(), and other server-side capabilities.

Sampling Result Object

  • result.content.text — the generated text
  • result.model — which model was used
  • result.stopReason — why generation stopped
  • result.role — always "assistant"

Use Cases

  • Auto-classify user input
  • Summarise retrieved documents
  • Generate structured output from raw data
  • Multi-step agent reasoning inside tools

2. Elicitation

Servers asking users for additional information during tool execution

CONCEPT

What is Elicitation?

💡

Definition: Elicitation allows an MCP server to pause mid-execution and ask the human user for more information. The server sends a structured form/question to the client, waits for user input, then continues with the provided data.

Elicitation Flow
User sends prompt to LLM
↓
LLM calls MCP tool
↓
Tool needs more info → elicit()
↓ sends form to client
User fills in the form / answers
↓ response back to server
Tool continues with user data
↓
Final result returned to LLM
Server pauses — waits for human — then resumes. True human-in-the-loop.

Why use Elicitation?

  • Tool needs info that wasn't in the original prompt
  • Collect confirmation before destructive actions
  • Ask for credentials or preferences at runtime
  • Interactive multi-step workflows
  • Avoid guessing — ask the user directly

Elicitation vs Sampling

  • Sampling → asks the LLM for more content
  • Elicitation → asks the human user for more input
  • Both pause the tool mid-execution
  • Both need client support to work
SCHEMA

Elicitation Request Structure

Request Fields

  • message — human-readable prompt shown to the user
  • requestedSchema — JSON Schema defining what fields to collect
  • Supports types: string, number, boolean, enum
  • Can mark fields as required

Response: action field

  • "submit" — user filled the form ✅
  • "cancel" — user dismissed the form ❌
  • "decline" — user refused to provide info 🚫

Response: content field

  • Dict of field names → user-entered values
  • Only present when action == "submit"
  • Server should always check action before using content
CODE

Elicitation in FastMCP

elicitation_server.pyPython
from fastmcp import FastMCP

mcp = FastMCP("ElicitationDemo")

@mcp.tool()
async def book_meeting(topic: str, ctx) -> str:
    """Book a meeting — asks user for date and attendees."""

    # Pause and ask user for additional details
    response = await ctx.elicit(
        message=f"Please provide meeting details for: {topic}",
        schema={
            "type": "object",
            "properties": {
                "date": {
                    "type": "string",
                    "description": "Meeting date (YYYY-MM-DD)"
                },
                "attendees": {
                    "type": "string",
                    "description": "Comma-separated email list"
                },
                "duration_mins": {
                    "type": "number",
                    "description": "Duration in minutes"
                }
            },
            "required": ["date", "attendees"]
        }
    )

    # Always check what the user did
    if response.action == "cancel":
        return "Meeting booking cancelled by user."

    if response.action == "decline":
        return "User declined to provide details."

    # action == "submit" — use the data
    data = response.content
    return (
        f"✅ Meeting booked!\n"
        f"Topic: {topic}\n"
        f"Date: {data['date']}\n"
        f"Attendees: {data['attendees']}\n"
        f"Duration: {data.get('duration_mins', 30)} mins"
    )


@mcp.tool()
async def delete_record(record_id: str, ctx) -> str:
    """Delete a record — ask for confirmation first."""

    response = await ctx.elicit(
        message=f"⚠️ Delete record {record_id}? This cannot be undone.",
        schema={
            "type": "object",
            "properties": {
                "confirm": {
                    "type": "boolean",
                    "description": "Confirm deletion"
                }
            },
            "required": ["confirm"]
        }
    )

    if response.action != "submit" or not response.content.get("confirm"):
        return "Deletion aborted."

    # Proceed with deletion
    return f"Record {record_id} deleted successfully."


if __name__ == "__main__":
    mcp.run()
🛡️

Best practice: Always check response.action before accessing response.content. If the user cancels or declines, content may be empty or None.

3. Authentication

Securing MCP servers — who can connect and how access is verified

CONCEPT

Why Authentication Matters

🔒

Without auth: Any client that knows your server URL can connect, call your tools, and access your data. For remote MCP servers, authentication is critical for production use.

Without Auth

  • Anyone can connect
  • Tools exposed publicly
  • Data can be read/modified
  • No usage tracking

With Auth

  • Only verified clients connect
  • Per-user access control
  • Secure data access
  • Audit trail possible

Local vs Remote

  • Local (STDIO) — less critical, runs on your machine
  • Remote (HTTP) — always use auth in production
TYPES

Authentication Methods

🔑 Bearer Token (API Key)

  • Simplest method — a secret string
  • Client sends in HTTP header: Authorization: Bearer <token>
  • Server validates the token on every request
  • Good for: machine-to-machine, dev use

Most Common

🔐 OAuth 2.0

  • Industry standard for user-facing apps
  • Issues short-lived access tokens
  • Supports scopes (fine-grained permissions)
  • MCP spec officially recommends OAuth 2.1

Production Standard

🍪 Session / Cookie

  • Server issues a session after login
  • Client stores session ID in cookie
  • Good for browser-based MCP clients

🔏 mTLS (Mutual TLS)

  • Both client & server present certificates
  • Highest security level
  • Common in enterprise environments
OAuth 2.0 Flow with MCP
MCP Client
(App)
→ auth request
Auth Server
(OAuth)
→ access token
MCP Client
→ token in header
MCP Server
MCP Server validates token with Auth Server before processing any tool calls
CODE

Implementing Authentication in FastMCP

auth_server.py — Bearer Token AuthPython
from fastmcp import FastMCP
from fastmcp.server.auth import BearerAuthProvider
import os

# Define valid API keys (in production: load from DB or env)
VALID_TOKENS = {
    "user-token-abc123":  {"user_id": "alice", "role": "admin"},
    "user-token-xyz789":  {"user_id": "bob",   "role": "viewer"},
}

async def verify_token(token: str):
    """Return user info dict if valid, raise exception if not."""
    if token not in VALID_TOKENS:
        raise ValueError("Invalid or expired token")
    return VALID_TOKENS[token]

# Attach auth provider to server
mcp = FastMCP(
    "SecureServer",
    auth=BearerAuthProvider(verify_token)
)

@mcp.tool()
async def get_secret_data(ctx) -> str:
    """Only authenticated users can call this."""
    # Access the authenticated user from context
    user = ctx.auth
    return f"Hello {user['user_id']}! Your role is: {user['role']}"


@mcp.tool()
async def admin_only_action(ctx) -> str:
    """Only admins can perform this action."""
    user = ctx.auth
    if user["role"] != "admin":
        return "❌ Access denied. Admin role required."
    return "✅ Admin action executed successfully."


if __name__ == "__main__":
    mcp.run(transport="http", host="0.0.0.0", port=8000)

auth_server_oauth.py — OAuth 2.1 (FastMCP Cloud)Python
from fastmcp import FastMCP
from fastmcp.server.auth import OAuthProvider

# FastMCP Cloud handles OAuth infrastructure for you
# Just enable it in deployment settings (Authentication: ON)
# The server automatically validates JWT tokens from clients

mcp = FastMCP(
    "OAuthServer",
    auth=OAuthProvider(
        issuer_url="https://auth.fastmcp.app",    # FastMCP's auth server
        required_scopes=["tools:read", "tools:write"]
    )
)

@mcp.tool()
async def protected_tool(data: str, ctx) -> str:
    user_id = ctx.auth.subject         # JWT "sub" claim
    scopes  = ctx.auth.scopes           # granted scopes
    return f"Processed for user {user_id}"
🚨

Never hardcode tokens in code. Always load from environment variables or a secrets manager: os.environ["API_TOKEN"] or python-dotenv.

CLOUD

FastMCP Cloud — Authentication Toggle

During Deployment Configuration

  • On the deployment page, toggle Authentication: ON
  • FastMCP Cloud automatically handles OAuth 2.1 token issuance
  • Clients must include a valid token in the Authorization header
  • Discoverable: OFF — hides your server from public listings
  • Server URL remains the same: https://your-name.fastmcp.app/mcp
✅ Auth ON (Production)
  • Only clients with valid token connect
  • Per-user access tracking
  • Safe to expose sensitive tools
  • Discoverable should be OFF
⚠️ Auth OFF (Dev/Testing)
  • Anyone with the URL can connect
  • No identity — all users anonymous
  • Fast to test, unsafe for production
  • Fine for public demo servers

Three Features — Side by Side

🤖 Sampling

  • Who asks: Server → LLM
  • What: AI text generation
  • When: Tool needs AI help mid-run
  • Method: ctx.sample()
  • Transport: STDIO only
  • Key field: messages, maxTokens

🙋 Elicitation

  • Who asks: Server → Human
  • What: Structured user input
  • When: Tool needs more info
  • Method: ctx.elicit()
  • Transport: STDIO / HTTP
  • Key field: schema, response.action

🔐 Authentication

  • Who checks: Server verifies Client
  • What: Identity + access control
  • When: Every connection/request
  • Method: Bearer / OAuth / mTLS
  • Transport: HTTP (remote servers)
  • Key field: ctx.auth, Authorization header
